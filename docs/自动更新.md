# 自动更新

`distill-studio` 现在已经接入了 Tauri v2 的 updater 基础能力，但当前仓库默认只完成了“应用内检查和安装”的代码接入，还没有把正式发布链路完全写死到仓库里。

## 当前实现方式

- GUI 左侧已经增加 `检查更新` 按钮。
- 点击后会：
  - 读取本地 updater 配置
  - 检查是否有新版本
  - 有更新时弹窗确认
  - 确认后下载、安装并重启应用
- 开发模式下 updater 配置文件位置：

```text
config/local/updater.json
```

- 发布版默认位置：

```text
<app config>/updater.json
```

说明：

- 仓库里保留了示例文件：

```text
config/examples/updater.example.json
```

- 当前 `src-tauri/tauri.conf.json` 里的 `plugins.updater` 只是最小空壳配置，用来让 updater 插件正常初始化。
- 应用运行时优先使用本地 `updater.json` 覆盖配置。
- 如果本地没有覆盖配置，则自动回退到发布构建时内置的 updater 配置。

## 推荐配置格式

```json
{
  "pubkey": "PASTE_TAURI_UPDATER_PUBLIC_KEY_HERE",
  "endpoints": [
    "https://github.com/AI4S-YB/distill-studio/releases/latest/download/latest.json"
  ]
}
```

## 最省事的发布方案

推荐直接使用：

- `GitHub Releases + latest.json`

这样不需要自己先买域名，只要更新地址是 `HTTPS` 即可。GitHub Releases 本身满足这个条件。

## 首次接通自动更新的最小步骤

### 1. 生成 updater 签名密钥

示例命令：

```bash
cd /Users/kentnf/projects/ai4s/distill-studio
npx tauri signer generate -w ~/.tauri/distill-studio.key
```

你会得到：

- 私钥：自己保管，不进 git
- 公钥：填到 `updater.json` 的 `pubkey`

## 2. 准备本地 updater 配置

复制示例文件：

```bash
cp config/examples/updater.example.json config/local/updater.json
```

然后填写：

- `pubkey`
- `endpoints`

## 3. 发布时生成 updater artifacts

当前仓库没有把 `createUpdaterArtifacts` 直接写死在基础 `tauri.conf.json`，这是有意保守处理，避免你现在的 `tauri build` 流程立刻被签名要求卡住。

现在仓库已经提供两种正式发布方式：

- 本地命令：

```bash
npm run tauri:build:release
```

说明：

- 现在本地 `tauri:build:release` 会优先自动读取：
  - `config/local/updater.json`
  - `~/.tauri/distill-studio.key`
  - `~/.tauri/distill-studio.key.pub`
- 如果私钥设置了密码，仍建议在本地环境变量或 `.env.local` 中提供：
  - `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`
- 现在 `release:preflight` 和 `tauri:build:release` 都会提前检查这个密码是否缺失，尽量在真正打包前就报清楚。

一个最直接的做法是把它写进本地 `.env.local`：

```text
TAURI_SIGNING_PRIVATE_KEY_PASSWORD="your-local-password"
```

- GitHub Actions：
  - `.github/workflows/release.yml`
  - tag 规则：`app-v*`

发布前建议先跑：

```bash
npm run release:preflight
```

如果本机已经登录 `gh`，可以直接同步当前仓库需要的 GitHub secrets：

```bash
npm run github:sync-secrets
```

发布时通常还需要提供：

- `TAURI_SIGNING_PRIVATE_KEY`
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`（如果你的私钥设置了密码）
- `TAURI_UPDATER_PUBLIC_KEY`

本地 release 配置文件由脚本生成，不应提交：

```text
src-tauri/tauri.release.conf.json
```

## 4. 上传 release 资产

发布时需要把 updater 相关产物一起上传到 GitHub Releases，至少包括：

- 安装包
- `latest.json`
- 对应平台的 updater artifact

如果你是本地手动打包，建议打包完成后立刻执行：

```bash
npm run release:assets
```

它会扫描 `target/release/bundle/`，把当前版本的安装包、`latest.json`、`.sig` 和压缩包整理成清单，减少人工漏传。

如果走 GitHub Actions，仓库里已经带了最小 workflow，会自动调用 `tauri-action` 创建 draft release 并上传产物。

建议在 GitHub 仓库里配置以下 secrets：

- `TAURI_SIGNING_PRIVATE_KEY`
- `TAURI_SIGNING_PRIVATE_KEY_PASSWORD`
- `TAURI_UPDATER_PUBLIC_KEY`

## 当前边界

- 现在已经具备“应用内检查更新”和“下载安装”的代码能力。
- 但如果没有配置 `updater.json`，按钮只会提示“自动更新尚未配置”。
- 这条提示只出现在：
  - 本地没有 `updater.json`
  - 且当前应用本身也不是带内置 updater 配置的正式发布构建
- 如果没有正式发布并上传 `latest.json` 和签名产物，自动更新不会真正命中新版本。
