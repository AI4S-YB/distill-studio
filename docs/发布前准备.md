# 发布前准备

这份清单用于在把 `distill-studio` 推到 GitHub 或制作桌面安装包前，快速确认仓库和发布物是干净的。

## 1. 先确认敏感信息不进入版本库

重点检查以下内容：

- `.env.local`
- `config/local/profiles/*.json`
- `config/local/gui.pipeline.json`
- `output/`
- `src-tauri/output/`
- `dist/`
- `target/`

建议命令：

```bash
cd apps/distill-studio
git status --short --ignored
```

预期结果：

- 上述本地文件应显示为 `!!`，表示已被 `.gitignore` 排除
- `git add .` 后不应把真实 API key、本地 profile、运行产物和构建产物加入暂存区

## 2. 扫描目录里是否还残留真实密钥

建议在发布前再做一次文本扫描：

```bash
cd apps/distill-studio
rg -n --hidden \
  --glob '!node_modules/**' \
  --glob '!target/**' \
  --glob '!.git/**' \
  'sk-[A-Za-z0-9]+' .
```

如果还有命中，需要逐个确认：

- 是否只是本地忽略文件
- 是否误写到了示例文件、README、构建产物或源码中

## 3. 确认发布版不依赖项目目录中的本地配置

当前项目已经按两种运行方式做了分离：

- 开发模式：继续使用项目目录中的本地配置，方便你当前机器开发
- 发布版本：自动切换到系统配置目录和系统数据目录

这意味着：

- 你本机当前使用方式不会被破坏
- 别人拿到发布版时，不会把你的项目目录当作可写配置目录

## 4. 构建前检查

建议至少执行：

```bash
cd apps/distill-studio
npm run build
cargo check --manifest-path src-tauri/Cargo.toml
```

如果准备实际生成桌面安装包，再执行：

```bash
cd apps/distill-studio
npm run tauri:build
```

## 5. 发布前人工检查

建议逐项确认：

- README 和文档已经切换为对外可读的中文说明
- 示例配置文件中没有真实 key
- 默认运行参数足够保守，首次体验不会直接把模型打爆
- GUI 能正常打开、保存配置、运行一次 smoke 测试
- 输出目录和结果打开按钮在当前系统上可用

## 6. 首次推送仓库

如果当前目录还没有初始化 git：

```bash
cd apps/distill-studio
git init
git add .
git commit -m "init: prepare distill-studio for release"
git remote add origin git@github.com:AI4S-YB/distill-studio.git
git branch -M main
git push -u origin main
```

如果远端已经存在内容，先 `git fetch origin` 并处理分支关系，不要直接强推。
