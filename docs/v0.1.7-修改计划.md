# v0.1.7 修改计划

最后更新：2026-04-24

## 1. 版本目标

`v0.1.7` 不追求继续扩展大功能，而是聚焦本轮 `v0.1.6` 测试中最影响真实使用闭环的几个问题，把“生成 -> 本地筛选 -> 上传平台 -> 后续更新”这一条主链路做得更稳。

本版本纳入范围的 5 个主题：

- 自动更新失败时更可恢复
- 输出目录改为用户可选
- `QA评测` 页联通检查结果与真实业务可用性一致
- `shard` 支持半成品落盘与续跑
- `浏览QA` 增加“快审”工作流，只上传保留问题

本版本明确不做：

- `模型试用` 空模型诊断
- `token` 消耗统计
- 任意旧目录导入为历史任务

## 2. 测试输入与设计结论

本轮测试里，对 `v0.1.7` 最直接有影响的结论包括：

- `v0.1.5 -> v0.1.6` 应用内更新已可用，但 GitHub 网络不稳时容易失败
- QA 生成本身可正常运行，但当前中断时仍会丢掉未完成 `shard` 的中间结果
- 输出目录当前固定托管，用户不能自己选择实际落盘位置
- `QA评测` 页存在“检查联通失败，但检查登录成功”的冲突状态
- 生成结果在上传前缺少一个正式的本地快速筛选步骤

因此 `v0.1.7` 的硬结论是：

- 不能再只依赖默认托管输出目录
- 不能再把“完整 `shard` 完成后一次写盘”当作唯一策略
- 不能再让上传动作默认发送所有问题
- 不能再让平台联通状态只依赖单一 `/health` 路由

## 3. 输出目录设计

### 3.1 目标

用户必须能够自己选择输出目录，而不是只能接受程序默认托管到系统目录或固定盘符。

但程序仍保留“按任务自动创建子目录”的行为，避免多任务互相覆盖。

### 3.2 界面设计

在设置页增加一组输出目录配置：

- `输出目录`
- `选择文件夹`
- `打开当前目录`
- `恢复默认`

用户可以选择任意本地文件夹，例如：

- `D:\\DistillStudioOutput`
- `E:\\科研数据\\QA输出`
- 桌面上的某个专用目录

### 3.3 行为规则

- 该配置表示“输出根目录”
- 每次新建任务时，程序仍在该根目录下自动创建批次子目录：
  - `topic-slug-timestamp`
- `浏览QA`、历史任务列表、继续任务，默认都只扫描当前输出根目录
- 如果目录不存在：
  - 允许自动创建
- 如果目录不可写：
  - 运行前直接阻止，并给出明确提示

### 3.4 兼容边界

- 修改输出目录后：
  - 新任务写入新目录
  - 旧任务不会自动迁移
- `继续当前任务` 只在当前输出根目录内匹配，避免跨目录误续跑

### 3.5 验收标准

- 用户能在设置页选择任意输出目录
- 新任务实际写入所选目录
- “打开输出文件夹”打开的是当前任务真实目录
- 历史任务和继续任务在新目录下仍可正常工作

## 4. 自动更新设计

### 4.1 问题判断

`v0.1.6` 已具备真正的发布链路和安装包更新能力，但当前应用内更新失败时，用户缺少清晰的恢复路径。

当前问题主要是：

- 更新检查对 GitHub 网络抖动敏感
- 提示信息还不够区分“无新版本”和“网络不可达”
- 用户失败后缺少直接跳转手动下载的入口

### 4.2 修改方向

- 放宽更新检查超时，或在短时间内增加一次重试
- 前端提示细分为三类：
  - 无新版本
  - 更新服务不可达
  - 下载/安装失败
- 在失败提示中提供手动下载入口，直接跳转当前 release 页面

### 4.3 验收标准

- 网络正常时，能稳定检查到更新
- GitHub 不可达时，提示不再模糊
- 用户能够直接转去 release 页手动下载

## 5. QA评测联通检查设计

### 5.1 问题判断

当前 `检查联通` 走的是：

- `GET /health`

而真实业务登录走的是：

- `POST /api/auth/login`

因此当前会出现一种误导场景：

- 健康检查失败
- 但登录成功
- 页面仍让用户以为平台不可用

### 5.2 修改方向

联通状态改为分层判定，而不是只盯一个健康检查路由：

1. 如果 `/health` 成功：
   - 显示联通正常
2. 如果 `/health` 失败，但登录成功：
   - 显示“健康检查失败，但业务可用”
3. 如果健康检查失败且登录也失败：
   - 显示联通失败

### 5.3 日志与文案

运行日志里需要分开记录：

- 健康检查失败
- 登录失败

不要再只给一个笼统的“平台连通失败”。

### 5.4 验收标准

- 不再出现“联通失败，但登录成功还显示不可用”的误导
- 用户能看出问题是在健康检查，还是鉴权本身

## 6. shard 半成品落盘与续跑

### 6.1 当前问题

目前仍是“一个 `shard` 全部完成后才写出一个 `shard_XXXX.json`”。

这会导致：

- 用户中途停止时，如果当前 `shard` 尚未完成，就没有任何中间结果
- 重新开始时，需要整片重跑这个未完成 `shard`

### 6.2 目标

允许当前 `shard` 在运行中保存已生成部分，下次继续时只补缺失内容，而不是整片重来。

### 6.3 修改方向

- 生成成功不再只在 `shard` 完成后统一落盘
- 每个 batch 成功后，立即把已完成记录写入当前 `shard` 对应文件或 checkpoint 文件
- 当前 `shard` 需要有明确状态：
  - `in_progress`
  - `completed`

### 6.4 浏览与续跑规则

- `浏览QA` 与历史任务页面必须能识别“部分完成的 shard”
- 续跑时：
  - 读取当前批次已有记录
  - 根据已有问题或 plan 标识去重
  - 只继续生成缺失部分

### 6.5 数据边界

- 不做“每条 QA 一个文件”
- 仍以 `shard` 为基本单位
- 但运行中允许存在“部分完成”的 `shard`
- 最终完成后再切状态为 `completed`

### 6.6 验收标准

- 中断后当前 `shard` 已完成部分不会丢
- 再次开始时不会整片重跑
- 浏览页能看到部分完成内容
- 未完成 `shard` 不会被误判为完整最终结果

## 7. 浏览QA 快审设计

### 7.1 目标

在正式上传平台前，给用户一个快速筛选问题质量的本地工作流。

重点不是做完整人工编辑器，而是让用户能够快速判断：

- 这个问题是否合适
- 是否需要修改问题文本
- 是否应该上传到平台

### 7.2 历史任务入口

在 `浏览QA` 历史任务列表里，每条任务的操作按钮扩展为：

- `浏览`
- `删除`
- `上传`
- `快审`

### 7.3 快审页面行为

点击 `快审` 后，进入该任务的快速审阅模式。

页面默认显示该任务第一个问题，并提供：

- 当前题号 / 总题数
- 问题文本
- 可编辑输入框
- `保存`
- `保留`
- `丢弃`
- `上一个问题`
- `下一个问题`

### 7.4 审阅状态

每个问题增加本地审阅状态：

- `未审`
- `保留`
- `丢弃`

其中：

- `丢弃` 只是本地标签
- 不真正删除原始生成数据
- 用户修改过的问题文本也要本地保存

### 7.5 本地落盘设计

不直接改写原始生成文件，例如：

- `generated/shard_*.json`

而是在任务目录下新增独立审阅文件，例如：

- `review_state.json`

建议至少记录：

- 问题唯一标识
- 审阅状态
- 编辑后的问题文本
- 更新时间

这样可以保证：

- 原始生成结果可追溯
- 审阅状态可逆
- 本地关闭应用后状态不丢

### 7.6 上传规则

当任务存在快审结果时，上传逻辑改为：

- 只上传状态为 `保留` 的问题

推荐明确约束：

- `未审` 不上传
- `丢弃` 不上传

如果一个任务中没有任何 `保留` 问题：

- 上传前直接提示“没有可上传的保留问题”

### 7.7 列表摘要

历史任务列表可增加快审摘要信息：

- `已审 x/y`
- `保留 n`
- `丢弃 m`

### 7.8 验收标准

- 用户可进入任意任务的快审模式
- 可编辑问题并保存
- 可逐题标记 `保留/丢弃`
- `上一个/下一个问题` 正常工作
- 审阅状态和编辑内容重启后仍保留
- 上传时只上传 `保留` 问题
- `丢弃` 不删除原始生成数据

## 8. 实施顺序建议

建议按下面顺序落地，避免前后依赖打架：

1. 输出目录改造
2. `shard` 半成品落盘与续跑
3. 浏览QA 快审与上传过滤
4. `QA评测` 联通检查重写
5. 自动更新提示与恢复路径增强

原因：

- 输出目录和任务扫描是后续历史任务、快审、续跑的共同基础
- `shard` 落盘策略会直接影响浏览和快审读到的数据
- 上传过滤必须建立在快审状态文件稳定之后
- 平台联通检查和自动更新属于相对独立的 UI/后端行为修正

## 9. 对外版本口径

`v0.1.7` 可以定义为一个“生成结果可控化”版本。

对用户最直观的变化有：

- 输出目录可自己选择
- 中断后不再轻易丢掉当前 `shard` 已生成内容
- 上传前可先做本地快审
- 上传时只上传保留的问题
- `QA评测` 平台状态判断更准确
- 更新失败时更容易恢复
