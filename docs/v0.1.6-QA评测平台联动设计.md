# v0.1.6 QA评测平台联动设计

最后更新：2026-04-23

## 1. 目标

`v0.1.6` 的目标不是在 `QA小灶` 内重做一套评测系统，而是把 `QA小灶` 和 `qaevaluate` 稳定打通，形成“生成 QA -> 上传平台 -> 上传者先自评 -> 需要时再送审”的闭环。

本版本聚焦 6 件事：

- 在 `QA小灶` 中显式展示当前版本号
- 在设置页中加入 `QA评测平台` 配置，且普通用户只看到一个平台地址
- 左侧菜单加入：
  - `QA评测`
  - `模型试用`
- 把当前“上传到任意 URL”的旧逻辑改成“面向 `qaevaluate` 正式接口的桥接上传”
- 上传鉴权改成普通可登录的专家账号，不在桌面端暴露管理员账号
- 上传后默认进入上传者自己的评测工作区，是否送给其他专家由上传者自己决定

## 2. 本次设计的硬结论

### 2.1 上传不能继续依赖管理员账号

当前 `qaevaluate` 现有的 `POST /api/admin/imports/push` 是管理员接口，桌面端如果继续走这条路，就必须把管理员账号放进应用设置里。这不符合产品安全边界。

因此 `v0.1.6` 的正式方向改为：

- 桌面端只允许用户填写自己的平台账号和密码
- 上传功能必须由平台新增专家可用的上传接口承接
- 不建议在桌面端保存管理员账号
- 不建议在桌面端引入“集成管理员账号”这种变体

结论：

- `QA小灶` 只存“上传者自己的平台账号”
- `qaevaluate` 负责补“专家上传、自评优先”的正式接口

### 2.2 设置页只暴露一个 `QA评测平台地址`

用户视角下，设置页不应同时出现：

- `QA评测页面地址`
- `QA评测接口地址`

这会让普通用户困惑。

因此 UI 只保留一个字段：

- `QA评测平台地址`

但桌面端内部仍保留两个运行时地址概念：

- `platformWebBaseUrl`
- `platformApiBaseUrl`

二者由同一个可见地址自动派生，不要求用户理解。

### 2.3 上传后默认进入“我的自评”，不是自动群发给其他专家

从用户角度，上传自己的 QA 后，第一关注点一定是：

- 我的 QA 有没有被平台正确接收
- 我自己能不能先检查和评估这一批 QA

因此默认流程应是：

- 上传
- 解析
- 进入上传者自己的评测工作区

而不是：

- 上传
- 自动分发给其他专家

是否送交其他专家，交给上传者自己决定。

### 2.4 正式桥接目标应是 JSON 接口，不是桌面端去兼容 multipart 文件上传

平台现有导入链路里，`multipart/form-data` 适合人工后台导入，不适合作为桌面应用的长期正式协议。

稳固方案应是：

- 平台补一个面向桌面端的专家 JSON 导入接口
- 桌面端直接按平台契约发送批次和 QA 行数据

不建议把桌面端长期设计成：

- 先导出平台专用 JSON 文件
- 再模拟后台上传文件

那样只是把后台表单流程搬进桌面端，不是正式系统对接。

结论：

- 长期正式方案：平台补专家 JSON push 接口，桌面端直接调用
- 如联调初期确实来不及补接口，可临时保留“导出平台 JSON 文件”的旁路，但不作为 `v0.1.6` 的正式主链路

### 2.5 元字段必须显式保留

`topic_name`、`subtopic`、`axis`、`qa_mode`、`provider`、`source_batch_name` 这些字段，不能再指望靠失败行的 `raw_payload_json` 顺带保存。

`v0.1.6` 首版必须明确：

- 平台成功导入后，也要保留这些元字段
- 平台需要有正式字段或 `metadata_json`

### 2.6 幂等性进入首版

同步场景里，重复点击上传、上传中断、重试上传、同一批次重复推送，都会真实发生。

因此 `v0.1.6` 必须把以下内容纳入首版：

- 批次级幂等
- 行级去重

这不是优化项，是协议基线。

### 2.7 打开平台页面时，浏览器仍可能要求再次登录

桌面端通过 API 登录拿到的 token，不会自动写入系统浏览器的 `localStorage`。

因此：

- 桌面端内上传成功
- 再点击“打开平台页面”

系统浏览器仍有可能先显示平台登录页。

这不是跳转失败，而是当前架构下的预期行为，设计和文案必须写清楚。

## 3. 当前事实

### 3.1 QA小灶 当前状态

当前浏览页已有“上传”按钮，但实现仍偏旧：

- 用户填写一个上传 URL
- 桌面端把当前批次 JSON 直接 `POST` 到该地址

问题在于：

- 这不是正式平台协议
- 没有稳定权限模型
- 没有标准批次状态和幂等保证
- 无法自然衔接平台评测工作区

### 3.2 qaevaluate 当前状态

本地开发目录：

- [qaevaluate](/Users/kentnf/projects/ai4s/qaevaluate)

本地接口文档：

- `http://127.0.0.1:8100/docs`

已确认的现状：

- 健康检查：
  - `GET /health`
- 登录：
  - `POST /api/auth/login`
- 现有 JSON 导入：
  - `POST /api/admin/imports/push`
- 现有导入列表：
  - `GET /api/admin/imports`
- 专家权限判定：
  - `require_expert` 允许 `expert` 与 `admin`
- 专家侧已有任务与历史能力：
  - `GET /api/expert/tasks`
  - `GET /api/expert/history`

这说明：

- 平台已经有专家账号体系
- 也已经有专家任务工作区
- 但“专家自己上传 QA 并进入自己的评测工作区”这条链路还没有正式接口

### 3.3 开发环境与生产环境地址形态不同

当前开发环境：

- Web 前端：`http://127.0.0.1:3100`
- API 后端：`http://127.0.0.1:8100`

当前生产环境预期：

- Web 页面：`http://10.33.105.218`
- API：`http://10.33.105.218`

所以“用户只看一个地址”与“系统内部要正确拆分 Web/API”必须同时满足。

## 4. 推荐总方案

### 4.1 结论

最稳的方案不是只改应用，也不是只改平台，而是“两边都改，但职责清晰”：

- `QA小灶`：
  - 改掉旧的任意 URL 上传方式
  - 改为面向 `qaevaluate` 正式协议上传
  - 在 UI 里只暴露一个平台地址和用户自己的账号密码
- `qaevaluate`：
  - 新增专家可用的上传与自评桥接接口
  - 保存来源元数据
  - 支持幂等
  - 为上传者创建自己的评测入口

这比“桌面端硬兼容管理员接口”更稳，也比“平台完全不改，桌面端模拟后台文件上传”更干净。

### 4.2 职责边界

`QA小灶` 负责：

- 生成批次
- 收集批次元信息
- 发起上传
- 展示上传结果
- 提供跳转入口

`qaevaluate` 负责：

- 鉴权
- 接收导入
- 解析导入
- 元数据持久化
- 幂等判定
- 给上传者建立自评工作区
- 在用户明确操作后再进入同行评审分发

## 5. 设置页设计

### 5.1 可见字段

设置页只新增一个类目：

- `QA评测平台`

其中只放以下字段：

- `QA评测平台地址`
- `QA评测用户名`
- `QA评测密码`

补充说明：

- `QA评测平台地址` 默认值：`http://10.33.105.218`
- 标注：`实验室内部使用`
- 用户名和密码是用户自己的平台账号，不是管理员账号

### 5.2 不向普通用户暴露的内部字段

以下字段可以存在于程序内部，但不进入普通设置页：

- `platformWebBaseUrl`
- `platformApiBaseUrl`

如果开发联调确实需要人工覆盖，建议：

- 只在开发模式显示隐藏高级项
- 或通过本地调试配置 / 环境变量覆盖

不要在正式产品设置页里长期暴露两个地址输入框。

## 6. 单地址派生规则

桌面端内部从 `QA评测平台地址` 派生出两个地址：

- `platformWebBaseUrl`
- `platformApiBaseUrl`

建议规则如下：

### 6.1 生产 / 内网发布

当用户填写：

- `http://10.33.105.218`

则内部直接派生：

- `platformWebBaseUrl = http://10.33.105.218`
- `platformApiBaseUrl = http://10.33.105.218`

### 6.2 本机联调

当用户填写的是：

- `http://127.0.0.1`
- `http://localhost`

则内部自动派生：

- `platformWebBaseUrl = http://127.0.0.1:3100`
- `platformApiBaseUrl = http://127.0.0.1:8100`

### 6.3 显式带端口的本机联调

为兼容实际调试，建议再支持：

- 输入 `http://127.0.0.1:3100`
  - 内部改写为：
    - `platformWebBaseUrl = http://127.0.0.1:3100`
    - `platformApiBaseUrl = http://127.0.0.1:8100`
- 输入 `http://127.0.0.1:8100`
  - 内部改写为：
    - `platformWebBaseUrl = http://127.0.0.1:3100`
    - `platformApiBaseUrl = http://127.0.0.1:8100`

用户看到的仍只是一个地址。

## 7. 平台侧需要补的正式接口

### 7.1 登录与健康检查

保留现有：

- `GET /health`
- `POST /api/auth/login`

桌面端先检查健康，再做登录。

### 7.2 新增专家上传接口

建议平台新增：

- `POST /api/expert/imports/push`

权限：

- `require_expert`

语义：

- 当前登录专家上传一整个 QA 批次
- 平台记录 uploader
- 平台创建或复用该批次

建议请求结构：

```json
{
  "name": "soybean-density-20260423",
  "source": "qa-xiaozhao",
  "external_batch_id": "soybean-density-20260423-1777",
  "technical_type_code": "cot_qa",
  "business_tag_codes": ["crop_breeding", "soybean"],
  "application_id": 1,
  "auto_parse": true,
  "create_self_review": true,
  "rows": [
    {
      "id": "qa_0001",
      "question": "...",
      "answer": "...",
      "context": "...",
      "difficulty": "medium",
      "source": "qa-xiaozhao",
      "model": "qwen-plus",
      "metadata": {
        "topic_name": "Soybean seed oil and protein improvement",
        "subtopic": "planting density",
        "axis": "experimental_design",
        "question_type": "reasoning",
        "audience": "researcher",
        "qa_mode": "cot",
        "provider": "openai-compatible",
        "provider_model": "qwen-plus"
      }
    }
  ]
}
```

### 7.3 新增专家导入查询接口

建议新增：

- `GET /api/expert/imports`
- `GET /api/expert/imports/{batch_id}`

用途：

- `QA小灶` 的 `QA评测` 页面查看最近上传结果
- 平台专家前端查看“我上传的批次”

### 7.4 新增送审接口

默认不自动送其他专家。

建议新增：

- `POST /api/expert/imports/{batch_id}/submit-for-peer-review`

语义：

- 上传者在完成自检后，主动把该批次送入正式同行评审分发

这样能符合你的产品判断：

- 上传者先自评
- 是否进入更大范围评测，由上传者自己决定

### 7.5 自评入口

首版不必强求再设计一套新页面名，但平台至少要能满足下面两件事：

- 上传成功后，这一批 QA 对应的自评对象归属于当前上传者
- 上传者可以在专家工作区看到并处理它们

实现上可二选一：

方案 A，上传时直接创建自评任务：

- `create_self_review = true`
- 平台自动把该批次拆成当前上传者的自评任务

方案 B，上传后再点一次开始自评：

- `POST /api/expert/imports/{batch_id}/start-self-review`

`v0.1.6` 更推荐方案 A，因为桌面端体验更顺。

## 8. 数据模型要求

### 8.1 批次表

平台侧建议在批次表中补充：

- `uploader_user_id`
- `external_batch_id`
- `source`
- `source_batch_name`
- `self_review_status`
- `peer_review_status`

唯一约束建议：

- `UNIQUE(source, uploader_user_id, external_batch_id)`

这样可以做到“同一上传者重复推送同一批次时不重复造批次”。

### 8.2 QA 行数据

平台侧建议在 QA 主表保留：

- `external_id`
- `metadata_json`
- `source`
- `source_model`

唯一约束建议：

- `UNIQUE(dataset_batch_id, external_id)`

### 8.3 元数据最小保留集合

`metadata_json` 首版至少保留：

- `topic_name`
- `subtopic`
- `axis`
- `question_type`
- `audience`
- `qa_mode`
- `provider`
- `provider_model`
- `grounding`
- `source_batch_name`

## 9. 字段映射

### 9.1 批次级映射

`QA小灶 -> qaevaluate`

- `name`
  - 使用批次标题，优先 `topic_name + 时间戳`
- `source`
  - 固定为 `qa-xiaozhao`
- `external_batch_id`
  - 使用本地批次目录 ID 或运行批次 ID
- `application_id`
  - 平台默认应用，或由后续平台返回可选项
- `technical_type_code`
  - `normal -> direct_qa`
  - `cot -> cot_qa`
- `business_tag_codes`
  - 根据领域标签映射
- `auto_parse`
  - 默认 `true`
- `create_self_review`
  - 默认 `true`

### 9.2 行级映射

- `id`
  - 对应 `record.id`
- `question`
  - 对应 `record.question`
- `answer`
  - 对应 `record.answer`
- `context`
  - 对应人类可读摘要文本
- `difficulty`
  - 对应 `record.difficulty`
- `source`
  - 固定为 `qa-xiaozhao`
- `model`
  - 对应生成模型名
- `metadata`
  - 对应来源元数据

### 9.3 context 建议内容

即使有 `metadata_json`，`context` 仍建议保留一段对人工友好的可读摘要：

```text
Topic: {topic_name}
Subtopic: {subtopic}
Axis: {axis}
Question Type: {question_type}
Audience: {audience}
QA Mode: {qa_mode}
```

## 10. 用户视角下的桌面端交互

### 10.1 浏览QA页的上传按钮

每个生成批次的 `上传` 按钮改为：

- 未配置平台地址或账号：
  - 提示先到设置页补齐
- 平台不可达：
  - 弹窗提示：`该功能仅限实验室内部网络使用`
- 未登录或 token 失效：
  - 自动尝试登录
- 登录成功：
  - 调用专家上传接口

成功提示建议明确写成：

- `批次已上传到 QA评测平台，并已进入你的自评工作区`

同时展示：

- `batch_id`
- `existing_batch`
- `self_review_status`

### 10.2 QA评测 菜单

`QA评测` 页面建议展示：

- 当前应用版本
- 平台地址
- 连通状态
- 登录状态
- 当前平台用户名
- 最近一次上传结果
- 打开平台评测页按钮

这个页面的定位不是重做整个平台，而是做：

- 联通检查
- 登录状态确认
- 上传结果回看
- 跳转入口

### 10.3 模型试用 菜单

`qaevaluate` 当前已有专家任务内 LLM 能力，但还不是完全独立的通用 playground。

因此 `v0.1.6` 的 `模型试用` 先做“平台入口版”：

- 检查平台连通性
- 检查登录状态
- 提供打开平台相应页面的入口

如果平台后续补独立试玩接口，再考虑在 `QA小灶` 内做对话式试用。

### 10.4 打开平台页面时的提示

按钮文案建议写成：

- `在浏览器中打开平台评测页`
- `在浏览器中打开模型试用页`

并明确提示：

- `如浏览器未登录平台，打开后可能先进入登录页`

## 11. 版本号展示

本版本要求用户一眼就知道自己正在使用哪个版本。

建议展示两处：

- 顶部状态区：`QA小灶 v0.1.6`
- 设置页标题区：`当前版本：v0.1.6`

实现原则：

- 直接读取 Tauri 应用版本
- 不在前端硬编码版本号

## 12. 联调顺序

### 第一阶段：平台补正式专家上传协议

`qaevaluate` 先完成：

- `POST /api/expert/imports/push`
- `GET /api/expert/imports`
- `GET /api/expert/imports/{batch_id}`
- 自评入口创建逻辑
- 元数据持久化
- 幂等约束

### 第二阶段：QA小灶 接正式协议

`QA小灶` 再完成：

- 设置页改成一个平台地址字段
- 上传按钮切到专家上传接口
- `QA评测` / `模型试用` 菜单
- 版本号显式展示

### 第三阶段：送审联动

双方最后再联调：

- `submit-for-peer-review`
- 上传后从自评转入同行评审

## 13. 最终结论

`v0.1.6` 的可实施方案已经明确：

- 不再让桌面端保存管理员账号
- 上传改为专家账号上传
- 设置页只显示一个 `QA评测平台地址`
- 系统内部自动派生 Web/API 地址
- 上传后默认进入上传者自己的自评工作区
- 是否送给其他专家，由上传者自己决定
- 正式对接目标应是平台新增的专家 JSON 上传接口
- 元数据保留和幂等都进入首版

这套方案的好处是：

- 用户理解成本低
- 安全边界更合理
- 联调路径清晰
- 后续可自然扩展到社区共建 QA、CoT QA、自评后送审等更复杂流程
